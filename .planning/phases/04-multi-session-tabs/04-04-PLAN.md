<task type="checkpoint:human-verify" gate="blocking">
  <what-built>
Multi-session terminal tabs with:
- Tab bar above the terminal area (SessionTabBar component)
- "+" button to spawn new sessions
- Status chips (colored dots): connecting/waiting/working/done/error
- Session names derived from first prompt sent
- Session history persisted to localStorage on tab close
- Protocol start messages carry stable sessionId (confirmed by passing unit tests)
  </what-built>
  <how-to-verify>
Start the app: npm run dev, open http://localhost:5173

**SESS-01 — Multiple sessions:**
1. Select a folder (or use the saved one)
2. Click the "+" button in the session tab bar
3. Verify a new session tab appears (named "Session 2")
4. Verify both sessions are independently connected to Claude CLI

**SESS-02 — Tab navigation:**
5. Click between tabs — verify the terminal switches without flickering or blanking
6. Type something in Session 1, switch to Session 2, switch back — Session 1's content is intact

**SESS-03 — Status indicators:**
7. Watch the status chip (colored dot) on the active session
8. While Claude CLI is processing output: chip should be orange (working)
9. After output stops for ~1.2s: chip should be grey (waiting)
10. Close a session (× button): chip was previously done/error in history

**SESS-04 — Session naming:**
11. In a fresh session (name shows "Session N"), type a message and send it
12. Verify the tab name changes to the first ~40 chars of your message
13. Send another message — verify the name does NOT change again

**SESS-05 — Session history persistence:**
14. Close a session tab (× button)
15. Hard reload the page (Cmd+Shift+R)
16. Open DevTools — Application — Local Storage — look for key slopmop_sessions_/your/cwd
17. Verify the closed session's metadata (name, status, closedAt) is present in the JSON

**SESS-06 — Protocol sessionId:**
18. Open DevTools — Network — WS — select the /ws connection
19. Click "Messages" tab
20. Find the "start" message sent on connection
21. Verify the JSON payload contains a "sessionId" field with a UUID value
  </how-to-verify>
